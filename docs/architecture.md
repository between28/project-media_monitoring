# 아키텍처 노트

## 왜 Python + SQLite + 데스크톱 앱인가

현재 저장소의 주 운영 경로는 `Python + SQLite + Windows 데스크톱 앱`입니다.

이 구성이 적합한 이유:
- 보도자료를 기준으로 세션을 나누어 추적하기 쉽습니다.
- 실행 이력, 기사 목록, 브리핑 결과를 로컬에 누적 저장하기 쉽습니다.
- 별도의 스프레드시트 기반 수동 작업 없이 바로 실행할 수 있습니다.
- 배포본을 폴더째 전달하면 Python이 설치되지 않은 Windows PC에서도 실행할 수 있습니다.

운영 개념은 `보도자료 1건 = 세션 1개`입니다.

## 자동화되는 부분

현재 자동화 범위:

1. `HWPX` 보도자료 텍스트 추출
2. 검색 쿼리 초안 생성
3. RSS / sitemap / Google News 수집
4. 제목/요약 기준 1차 필터링
5. 중복 판정
6. 정책 관련도 점수화
7. 프레임 분류
8. 브리핑 초안 작성
9. 참고기사 목록 CSV / Markdown 생성

## 사람이 최종 검토하는 부분

사람이 개입해야 하는 부분:
- 자동 추출된 검색 쿼리 보정
- 핵심 키워드 입력 또는 조정
- 기사 누락 / 과포함 확인
- 브리핑 문구 최종 확정

이 프로젝트는 완전 자동화보다 `검색 규칙 초안 자동 생성 + 사람 승인`에 무게를 둡니다.

## 데이터 흐름

1. 보도자료 `HWPX` 입력
2. `queries.auto.json` 생성
3. 사용자가 `queries.manual.ini` 또는 GUI 편집창에서 최종 검색 규칙 확정
4. 수집 결과를 `session.sqlite3`에 저장
5. 대표 기사 목록과 브리핑을 `sessions/<session_id>/outputs/`에 생성

## 향후 확장

### GitHub Actions

가능한 확장:
- 정해진 시각의 CLI 실행
- 세션 산출물 자동 보관
- 설정 검증 및 테스트 자동화

### 네트워크 저장소 / 공유 드라이브

현 배포 방식은 로컬 중심이지만, 세션 폴더를 공유 드라이브에 저장하도록 바꾸면 협업 검토도 가능합니다.

### 고도화 대상

- 기사 본문 추출 정확도 개선
- 포털 래퍼 링크의 원문 URL 복원 강화
- 언론사명 추출 규칙 보강
- 세션 간 비교 리포트

핵심은 단순합니다. 현재 구조는 `사람이 수정 가능한 검색 규칙`과 `세션 단위 저장`을 중심으로 유지하고, 나머지 자동화는 그 위에서 확장합니다.
